Система, состоит из трёх частей: база данных MySQL, PHP API на сервере и код для Arduino UNO (GSM шлюз) с SIM800L. Реализовано хранение до 10000 номеров на SPI Flash, синхронизация с сервером (проверка хеша, загрузка обновлений), обработку входящих звонков, управление открытием/закрытием ворот через реле (выходы 8 и 9), приоритет первой группы, задержку для второй группы, логирование событий, отправку SMS команд с сайта, сохранение SMS в БД, управление данными через UI (на сервере). Токен для безопасности.

Структура БД и API.

База данных MySQL:

Таблица номеров (whitelist): id, номер, группа (0 - первая без задержки, 1 - вторая с задержкой), активен (bool), дата создания/обновления. Можно хранить хеш всей таблицы для быстрой проверки изменений.

Таблица логов событий: id, тип события (звонок, открытие, закрытие, ошибка и т.д.), номер (если применимо), группа (если применимо), статус, детали, timestamp.

Таблица SMS команд: id, номер отправителя, команда, статус обработки, timestamp.

Таблица настроек шлюза: ключ, значение (например, хеш данных, последняя синхронизация, токен для доступа к API и т.д.) - но настройки могут храниться локально на шлюзе, но для синхронизации можно использовать отдельную таблицу на сервере для конфигурации шлюзов? Возможно, каждый шлюз имеет свой ID и токен, и свои настройки. У нас один шлюз, но на будущее можно предусмотреть.

Упростим: есть один шлюз, его настройки хранятся в таблице gateway_config: ключ, значение. Например, token (для аутентификации запросов от шлюза), data_hash (последний известный хеш номеров), sync_interval (интервал синхронизации). На самом деле интервал у нас фиксированный 5 минут, но можно хранить.

PHP API должен предоставлять эндпоинты:

GET /api/get_hash - возвращает текущий хеш данных (MD5 от конкатенации всех номеров с группами?).

GET /api/get_data?hash=<client_hash> - возвращает данные (номера с группами), если хеш клиента не совпадает с текущим (или всегда возвращать данные с хешем). Токен в заголовке Authorization: Bearer <token>.

POST /api/log - для отправки логов с шлюза (чтобы сохранять в БД). Принимает JSON с массивом событий.

POST /api/sms - для отправки SMS команд (отправка SMS через SIM800L с сайта). Шлюз должен периодически опрашивать сервер на наличие новых SMS для отправки? Или сервер отправляет команду на шлюз? У нас шлюз инициирует связь (HTTP запросы). Значит, шлюз может раз в некоторое время (например, раз в минуту) запрашивать команды. Или можно реализовать push через длинные опросы, но проще polling. Создадим эндпоинт GET /api/commands, который возвращает список команд (например, отправить SMS на определенный номер с текстом, или закрыть ворота). После выполнения шлюз подтверждает выполнение через POST /api/command_ack.

POST /api/update_numbers - для администратора (из UI) для добавления/редактирования номеров. Но это отдельно, можно через веб-интерфейс.

UI для управления номерами: добавление, удаление, изменение группы, просмотр логов, отправка SMS команд. Это отдельная задача, но мы должны предоставить код PHP для API и, возможно, простой интерфейс.

SQL скрипты для создания таблиц:

Таблица номеров:
CREATE TABLE numbers (
id INT AUTO_INCREMENT PRIMARY KEY,
phone_number VARCHAR(20) NOT NULL UNIQUE,
group_id TINYINT NOT NULL DEFAULT 0 COMMENT '0 - без задержки, 1 - с задержкой',
active BOOLEAN DEFAULT TRUE,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

Таблица логов:
CREATE TABLE logs (
id INT AUTO_INCREMENT PRIMARY KEY,
event_type VARCHAR(50) NOT NULL,
phone_number VARCHAR(20),
group_id TINYINT,
status VARCHAR(50),
details TEXT,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Таблица SMS команд, отправленных с сайта (для последующей отправки через шлюз):
CREATE TABLE sms_commands (
id INT AUTO_INCREMENT PRIMARY KEY,
phone_number VARCHAR(20) NOT NULL,
message TEXT NOT NULL,
status ENUM('pending', 'sent', 'failed') DEFAULT 'pending',
attempts INT DEFAULT 0,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
sent_at TIMESTAMP NULL
);

Таблица для хеша данных (можно хранить в отдельной таблице или вычислять на лету). Для простоты будем вычислять хеш по запросу. Но для синхронизации нужна возможность быстро получить хеш. Можно создать таблицу metadata с ключом 'numbers_hash' и обновлять его при изменениях через триггеры или в приложении. Я предлагаю создать таблицу gateway_config:
CREATE TABLE gateway_config (
config_key VARCHAR(50) PRIMARY KEY,
config_value TEXT
);
И вставить запись для токена: INSERT INTO gateway_config (config_key, config_value) VALUES ('api_token', 'some_random_token');
А также для хеша: 'numbers_hash' будет обновляться при изменении таблицы numbers (через триггер или из кода).

Но чтобы не усложнять, хеш можно вычислять каждый раз по запросу (SELECT MD5(GROUP_CONCAT(phone_number, group_id ORDER BY phone_number)) FROM numbers WHERE active=1). Это может быть медленно для 10000 записей, но терпимо. Или можно использовать триггер на обновление хеша в отдельной таблице.

Я создам триггеры для обновления хеша в таблице metadata при изменении numbers.

Теперь PHP API. Будем использовать простой PHP без фреймворков. Обработка запросов, проверка токена (из заголовка Authorization). Реализуем эндпоинты:

/api/get_hash (GET) - возвращает JSON: {"hash": "..."}

/api/get_data (GET) - принимает параметр hash (необязательный). Если передан hash и совпадает с текущим, возвращаем {"changed": false}. Иначе возвращаем {"changed": true, "data": [{"phone": "...", "group": 0}, ...], "hash": "..."}

/api/log (POST) - принимает JSON массив логов, вставляет в таблицу logs.

/api/commands (GET) - возвращает список pending SMS команд (максимум N, например 10). После выполнения шлюз отправит подтверждение.

/api/command_ack (POST) - помечает команду как выполненную (статус sent или failed).

Также для UI нужны страницы: список номеров с возможностью добавления/редактирования, просмотр логов, отправка SMS.

Arduino код. Задача: на UNO с SIM800L, SPI Flash (предположим, W25Q64 или подобный, 8MB), часы реального времени? Для логов и задержек нужно время. У SIM800L есть RTC, можно получать время из сети. Но проще использовать millis() для отсчета задержки, но для логов нужно абсолютное время. Можно получать время из сети при каждом событии или при старте и синхронизировать. Для простоты, будем использовать внутренний счетчик времени с момента включения, а для логов на сервере будет timestamp сервера. Но нужно логировать события на шлюзе до отправки на сервер, поэтому можно хранить локальные логи с временем от внутреннего таймера (uint32_t), а при отправке на сервер добавить текущее время сервера или просто полагаться на время сервера при получении лога. Лучше, чтобы шлюз передавал временную метку (время от загрузки), а сервер добавлял свое время получения. Или шлюз мог бы запрашивать время через NTP или из сети оператора. Но для упрощения, пусть сервер фиксирует время получения лога. Тогда логи на шлюзе могут содержать только тип и номер, а время события будет определяться сервером при получении (но это будет время получения, а не события, что может быть не точно при задержках). Если критично, можно синхронизировать часы. SIM800L поддерживает AT+CCLK? Можно получить время из сети. Будем использовать это.

Основные компоненты кода Arduino:

Инициализация SIM800L (SoftwareSerial).

Инициализация SPI Flash.

Загрузка номеров из Flash в оперативную память. 10000 номеров - это много для UNO (2KB RAM). Нужно хранить на Flash и искать по номеру при звонке. SPI Flash позволяет читать произвольные адреса. Можно организовать хранение в виде списка записей фиксированной длины (например, 20 байт на номер + 1 байт группа = 21 байт). 10000 * 21 = 210000 байт, что помещается в 8MB Flash. Но для поиска придется читать последовательно, что медленно. Можно использовать хеш-таблицу или индекс. Но для 10000 и скорости поиска при звонке (один звонок в минуту) последовательный поиск с чтением с SPI Flash (скорость ~10-20 MB/s) может занять доли секунды: чтение 210KB займет ~0.01-0.02 сек, что допустимо. Можно организовать бинарный поиск, если хранить номера отсортированными. Тогда нужно поддерживать сортировку при загрузке с сервера. Сервер присылает номера, мы их сортируем и сохраняем на Flash. При поиске выполняем бинарный поиск по номеру. Это быстрее.

Таким образом, данные на Flash хранятся как отсортированный список структур: номер (как строка или как uint64_t, но номер может начинаться с +, до 15 цифр, можно хранить как строку фиксированной длины 16 байт + нулевой терминатор, или как uint64_t, но тогда нужно преобразовывать). Проще хранить как строку 15 цифр + 1 байт группа. Итого 16 байт на запись + возможно разделитель. Для 10000 - 160KB, хорошо.

Алгоритм:

При старте: инициализация SIM800, SPI Flash. Чтение заголовка (хеш данных, количество записей) с Flash. Если заголовок пустой, устанавливаем начальные значения.

Подключение к GPRS (если нужно для HTTP).

Запуск цикла: обработка входящих звонков (обнаружение по AT+CLCC или по индикации RING), обработка входящих SMS (для команд закрытия и других), периодическая синхронизация с сервером (раз в 5 минут).

Синхронизация: отправляем GET /api/get_hash с токеном. Получаем хеш. Сравниваем с локальным. Если отличается, запрашиваем GET /api/get_data, получаем новый список номеров, сортируем, записываем на Flash, обновляем локальный хеш. Также получаем команды (GET /api/commands) для отправки SMS.

Обработка звонка: определяем номер (AT+CLCC или из строки RING). Ищем в списке номеров бинарным поиском. Если найден, проверяем группу. Если группа 0 (без задержки) -> сразу открываем ворота (устанавливаем высокий уровень на пине 8 на 1 сек?). Если группа 1 (с задержкой) -> запускаем таймер на 5 минут. Во время ожидания проверяем, не поступил ли звонок с группы 0. Если поступил, отменяем таймер и открываем сразу. Если таймер сработал, открываем. Также нужно учитывать, что если ворота уже открыты, возможно не надо открывать повторно? По задаче не уточнено. Будем считать, что открытие происходит независимо от состояния.

Обработка SMS: при получении SMS (AT+CNMI) читаем текст. Если команда "ЗАКРЫТЬ" (или определенный текст), закрываем ворота (высокий на пине 9). Можно также реализовать команды от определенных номеров. Но по заданию: "Закрытие по SMS или команде с сайта." Значит, любая SMS с командой закрытия с любого номера? Или только с определенных? Уточним: вероятно, закрытие может быть инициировано либо SMS (с любого номера), либо с сайта. Но для безопасности, лучше разрешить закрытие только с определенных номеров (администраторов). Однако в задании не сказано. Возможно, подразумевается, что шлюз принимает SMS от любого, но тогда это небезопасно. Но мы можем реализовать проверку номера отправителя по той же таблице? Но в таблице только для открытия. Лучше добавить отдельный список для команд или использовать общий. Для упрощения, предположим, что любая SMS с текстом "CLOSE" (или "ЗАКРЫТЬ") инициирует закрытие. А для безопасности можно использовать токен в SMS? Но это неудобно. Я думаю, в реальности используют отдельные номера администраторов. Но по заданию не указано. Оставим так: любая SMS с командой "CLOSE" (регистр не важен) приводит к закрытию. При этом событие логируется. Если нужно ограничить, можно легко добавить проверку по списку администраторов, который можно хранить на Flash отдельно или в той же таблице с флагом admin.

Также нужно обрабатывать команды с сайта: они приходят через API /api/commands. Это могут быть команды на отправку SMS (чтобы инициировать закрытие через SMS) или команда на закрытие напрямую? Возможно, сайт может отправить команду шлюзу на закрытие ворот (без SMS). Для этого добавим тип команды: "close_gate". Тогда шлюз, получив команду, выполняет закрытие. Или команда на отправку SMS: "send_sms" с номером и текстом.

В API /api/commands будем возвращать список команд в формате JSON: [{"id": 1, "type": "send_sms", "phone": "123", "message": "CLOSE"}, {"id": 2, "type": "close_gate"}]. Шлюз выполняет и отправляет подтверждение.

Теперь код Arduino. Нужно минимизировать использование памяти. Будем использовать буферы для чтения из Flash и для HTTP запросов. SIM800L требует работы с AT командами. Используем библиотеку SoftwareSerial. Напишем свой парсинг AT команд.

Основные функции:

setupSIM800() - инициализация, проверка регистрации в сети.

sendAT() - отправка команды и ожидание ответа.

httpGet() - выполнение HTTP GET запроса.

httpPost() - для отправки логов.

readIncomingCall() - проверка входящего звонка.

readSMS() - чтение новых SMS.

findNumber() - бинарный поиск номера в Flash.

updateNumbersFromServer() - получение данных, сортировка и запись на Flash.

processCall() - обработка звонка.

processSMS() - обработка SMS.

checkCommands() - опрос команд с сервера.

logEvent() - добавление события в локальный буфер логов и отправка на сервер (периодически или сразу). Чтобы не отправлять каждый лог отдельно, будем накапливать и отправлять пачкой раз в минуту или при превышении размера.

Для хранения логов на шлюзе до отправки можно использовать небольшой буфер в RAM (несколько записей) и сбрасывать на сервер. При отключении питания логи потеряются, но это допустимо? Можно также хранить на Flash, но усложнит код. Для простоты, будем отправлять логи сразу после события, если есть GPRS соединение. Если нет, можно накапливать в буфере и пытаться отправить позже. Но для UNO памяти мало, поэтому лучше отправлять сразу.

Теперь про SPI Flash. Будем использовать библиотеку SPI.h для работы с флешкой. Драйвер для конкретной микросхемы нужно будет написать или использовать готовую (например, Adafruit SPIFlash). Но для простоты будем использовать низкоуровневые функции: чтение/запись страниц, секторов.

Структура данных на Flash:

Заголовок (первые несколько байт): магическое число (2 байта), версия (1 байт), количество записей (2 байта, до 10000), хеш данных (16 байт MD5? но можно 32 байта строки hex), контрольная сумма.

Затем идут записи: каждая запись фиксированной длины: номер (15 символов + нулевой терминатор = 16 байт), группа (1 байт). Итого 17 байт. Но лучше хранить номер как строку с завершающим нулем для удобства использования strcmp. Тогда длина записи 17 байт. Для 10000 записей - 170KB. Плюс заголовок, всего около 200KB. Можно использовать 256 байт на запись? Нет, избыточно.

Для бинарного поиска нужно читать записи по индексу. Для этого нужно знать смещение каждой записи: заголовок + (index * record_size). Тогда читаем запись по нужному индексу, сравниваем номер.

При получении новых данных с сервера (JSON массив), нужно отсортировать по номеру. Сортировка на Arduino: можно использовать qsort, но для 10000 элементов это займет много времени и памяти (нужен массив указателей). Но мы можем не сортировать на Arduino, если сервер присылает уже отсортированный список. Это оптимально: сервер при формировании ответа сортирует номера и присылает в порядке возрастания. Тогда Arduino просто записывает их последовательно. Это идеально.

Таким образом, API /api/get_data возвращает отсортированный по номеру массив.

Теперь реализация бинарного поиска:

Читаем заголовок, получаем количество записей N.

Устанавливаем low=0, high=N-1.

Пока low <= high:
mid = (low+high)/2
читаем запись по смещению header_size + mid*record_size
сравниваем номер с искомым (strcmp)
если равно, возвращаем группу
если меньше, low = mid+1
если больше, high = mid-1

Для чтения записи нужно читать 17 байт из Flash. Используем библиотеку SPI.

Теперь код для SIM800L: обрабатываем входящие звонки. Есть два способа: использовать URC (Unsolicited Result Code) "RING", и затем запросить информацию о звонке через AT+CLCC. Мы можем в основном цикле проверять наличие входящего вызова, анализируя приходящие от SIM800 строки. Для этого нужно читать данные из SoftwareSerial и искать "RING". При обнаружении RING, можно сразу ответить (ATH?) чтобы сбросить вызов? По заданию: "Открывать по звонку только." Значит, мы должны ответить на звонок? Обычно для таких систем используется пропущенный звонок: шлюз сбрасывает вызов, определяет номер и открывает. Но в задании сказано "по звонку", возможно, имеется в виду, что шлюз принимает звонок, определяет номер и открывает, но сам разговор не нужен. Обычно используют режим "игнорировать звонок", т.е. после определения номера сбрасывают. Либо можно ответить и сразу повесить. Для определения номера без ответа можно использовать AT+CLCC, но для этого нужно, чтобы SIM800 сообщал о входящем звонке с номером. В URC "RING" номер не всегда приходит, зависит от оператора. Можно настроить SIM800 на выдачу номера при звонке через AT+CLIP=1. Тогда при RING будет строка +CLIP: "<number>",... Так и сделаем.

После получения номера, ищем его. Если найден, обрабатываем по группам. Затем сбрасываем вызов командой ATH.

Для управления выходами: pinMode(8, OUTPUT); digitalWrite(8, HIGH); delay(1000); digitalWrite(8, LOW); для открытия. Аналогично для закрытия пин 9.

Теперь логирование: при каждом значимом событии (звонок, открытие, закрытие, ошибка) формируем строку лога и отправляем на сервер POST /api/log. Можно отправлять сразу, но чтобы не блокировать основной цикл, можно делать это асинхронно. Но для простоты будем отправлять сразу и ждать ответа.

Важно: SIM800L может быть занят отправкой HTTP, в это время не обрабатывать звонки. Но звонок может поступить в любой момент. Надо организовать неблокирующий режим: в основном цикле проверяем наличие данных от SIM800, если есть - читаем построчно, анализируем. Если нужно отправить HTTP, отправляем, но при этом продолжаем проверять входные данные (например, прерывая отправку по таймауту). Это сложно. Можно использовать конечный автомат. Учитывая, что UNO однозадачный, придется делать все последовательно, но с тайм-аутами. Например, при обнаружении звонка, приоритет выше, и мы прерываем текущую операцию (например, синхронизацию) и обрабатываем звонок. Но HTTP запросы могут быть длительными. Можно использовать переходы между состояниями. В целом, для данного проекта, думаю, можно реализовать простой цикл, в котором последовательно опрашиваем звонки, SMS, проверяем таймеры, и периодически запускаем синхронизацию и проверку команд. Если во время HTTP приходит звонок, данные в буфере SIM800 будут накапливаться, и мы сможем обработать их после завершения HTTP, но звонок может быть пропущен. Чтобы минимизировать, можно делать HTTP запросы короткими (таймаут 10-15 сек) и часто проверять наличие данных во время ожидания. Это возможно, если реализовать чтение с таймаутом и проверку буфера.

В коде будем использовать функции с таймаутами и возможностью прерывания по приходу данных. Например, при отправке AT команды и ожидании ответа, можно в цикле читать по байту и проверять, не пришла ли строка RING. Это сложно, но можно.

Упростим: будем опрашивать звонки в начале каждого цикла, перед другими операциями. Если звонок есть, обрабатываем, иначе идем дальше. Синхронизацию делаем раз в 5 минут, но если идет обработка звонка, можно отложить синхронизацию. Такой подход допустим.

Триггер для обновления хеша при изменении numbers.

Но можно обновлять хеш в PHP при каждом изменении через UI, тогда триггер не нужен. Но для согласованности, лучше триггер. Однако триггеры в MySQL могут быть сложны при массовых обновлениях. Я предлагаю вычислять хеш динамически в API, так как это проще. Но тогда при каждом запросе хеша будет выполняться GROUP_CONCAT на 10000 строк, что может быть нагрузкой. Можно кешировать хеш в отдельной таблице и обновлять его через триггер или при каждом изменении из кода. Выберем вариант с триггером.

Создадим таблицу metadata с двумя полями: key VARCHAR, value TEXT. Вставим запись для хеша: ('numbers_hash', ''). Создадим триггер AFTER INSERT, UPDATE, DELETE на таблице numbers, который пересчитывает хеш и обновляет metadata. Для пересчета хеша используем SELECT MD5(GROUP_CONCAT(phone_number, group_id ORDER BY phone_number)) FROM numbers WHERE active=1. Это будет выполняться при каждом изменении, что не очень эффективно для частых изменений, но для администрирования через UI это приемлемо.

Альтернатива: вычислять хеш при каждом запросе от шлюза и кешировать в PHP на несколько минут. Но тогда возможны рассинхронизации. Я думаю, триггер подойдет.

Эндпоинты:

action=get_hash: возвращает {"hash": hash}

action=get_data: если передан параметр hash и равен текущему, возвращаем {"changed": false}. Иначе выбираем все active номера (phone, group) сортируем по phone, формируем JSON: {"changed": true, "hash": hash, "data": [{"phone": "...", "group": id}]}

action=log: принимает POST JSON массив логов, вставляет в таблицу logs (каждый лог с типом, номером, группой, статусом, деталями). Возвращает {"status": "ok"}

action=commands: возвращает список pending команд из sms_commands (до 10). Формат: [{"id": id, "type": "send_sms", "phone": phone, "message": message}] Также можно добавить команду "close_gate" как специальный тип. Для простоты, пусть команда "close_gate" будет с type="close_gate" и без дополнительных полей. Тогда шлюз ее выполнит.

action=command_ack: принимает POST {"id": id, "status": "sent/failed", "details": ...} помечает команду как выполненную.
